很多产品的“S3 支持”意味着定期把本地文件备份到 Bucket。对 ZPan 来说，S3 不是备份目的地，而是文件唯一的数据平面。这个差异决定了上传路径、数据库职责、部署成本和迁移方式。

## 控制平面不保存文件字节

ZPan 的数据库保存用户、空间、目录、对象元数据、分享、配额和任务。Bucket 保存文件内容。上传时，服务端先验证权限并创建上传会话，再把预签名地址交给浏览器。完成后，浏览器回到 ZPan 确认对象状态。

于是应用服务器不需要与用户文件大小成比例的带宽。一个大文件不会先进入 Worker 或 Node 进程，再被复制到 S3。对象存储承担传输、分片和分发，控制平面保持轻量。

## 数据库与 Bucket 必须保持一致

S3 原生不代表可以绕过 ZPan 随意修改 Bucket。对象 Key 是实现细节，目录、名称、权限和回收站状态在数据库中。直接从云控制台改名或删除对象，会让控制平面与数据平面失去一致性。

备份也因此需要两部分：数据库备份保护目录和权限，Bucket 版本控制或复制保护文件内容。只备份其中一边都不能完整恢复站点。

## CORS 是架构的一部分

直传发生在浏览器，所以 Bucket 必须允许 ZPan 的正式 Origin 发起 `PUT`、`POST`、`GET` 和 `HEAD`，并暴露 `ETag`。这不是可选优化，而是数据路径的必要条件。

最常见的故障模式是后台可以保存存储配置，用户上传却失败。原因往往不是 ZPan 无法连接 S3，而是浏览器预检被 Bucket 拒绝，或者 Endpoint 只在 Docker 内网可解析。

ZPan 的连接测试会从当前浏览器实际上传一个临时对象，再清理它。只有这条测试通过，才能证明服务端签名、浏览器网络、CORS 和 Bucket 权限一起工作。

## S3 兼容不等于所有实现完全相同

AWS S3、R2、B2、Tigris、MinIO 和 RustFS 共享核心协议，但 Region、Endpoint、Path Style、公开域名和部分分片行为仍有差异。ZPan 提供常见服务模板，同时保留 Force Path Style 等选项。

站长仍应使用专用最小权限凭证，并阅读当前存储商的 CORS 与签名文档。所谓兼容，意味着产品可以在一个稳定协议上适配差异，而不是假设所有服务毫无差别。

## 可迁移性来自清晰边界

文件以普通对象形式保存在你选择的 Bucket 中，不需要先从某种专有归档格式导出。但完整迁移仍需要同时处理对象与数据库映射，并验证访问域名、缓存和路径。

清晰边界带来的价值不是“一键随便搬”，而是你知道每一部分在哪里：应用可以换运行平台，数据库可以在 SQLite 家族中迁移，Bucket 可以按计划复制，公开域名则保持稳定。

这就是 ZPan 所说的 S3 原生：不是在功能列表里多一个 Driver，而是让对象存储负责文件，让 ZPan 专注于把文件变得可管理、可分享和可运营。
