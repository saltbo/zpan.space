开发者和内容创作者通常不缺一个能保存图片的 Bucket，真正缺的是一条足够顺手、又能长期掌控的发布路径：截图后立即上传、自动生成目录、复制稳定链接，还能在需要时撤销某台设备的权限。

ZPan 2.4 把图床作为一等场景。它没有另建一套存储，而是复用 ZPan 已有的用户、工作空间、S3 直传和权限能力，在上面增加图片配置、图库、API Key、自定义域名与工具集成。

## 每个工具使用独立 API Key

PicGo、PicList、uPic、ShareX、Flameshot 和自定义脚本都可以连接 ZPan。建议每个设备或工具创建单独的 API Key，而不是共用账号密码。Key 只在创建时显示明文，服务端保存 Hash；某台设备不再使用时，可以单独撤销。

工具集成页面会生成对应配置，避免手工拼接 Header、表单字段和响应路径。Key 只在浏览器当前页面用于生成配置，不会为了方便而再次上传到服务器保存。

## 路径规则决定资产是否可维护

图床用久之后，最大的麻烦往往不是上传，而是对象命名混乱。2.4 支持按年、月、文件名等变量组织路径。博客图片可以按文章或日期归档，截图可以进入独立前缀，避免所有对象堆在 Bucket 根目录。

规则应在正式使用前确定。改变规则只影响新上传，不会自动迁移旧对象；依赖旧 URL 的文章也不应该被批量改名。

## 使用自己的访问域名

自定义域名让公开链接不再暴露存储商地址，也便于以后更换底层 Bucket。Cloudflare Workers 可以协助管理相关域名能力，Docker 部署则由站长在 Caddy、Nginx 或其他代理中配置 DNS 与 HTTPS。

域名只是访问层，原始文件仍在配置的 S3 存储中。切换域名前应先验证新旧地址的缓存、Content-Type 和公开访问策略，并为旧链接保留迁移或重定向方案。

## 防盗链与限制

图床可以设置 MIME、大小和 Referer 规则。Referer 只能作为减少普通盗链的手段，不能当作强身份认证；浏览器或客户端可能不发送它，也可以伪造它。真正私密的图片不应该依赖公开图床 URL。

## 从第一张测试图开始

站长先完成对象存储与 CORS 测试，再由用户启用图床、选择存储与路径规则、创建 API Key，最后在工具中上传一张测试图。检查返回 URL、Content-Type、缓存和删除行为都符合预期后，再迁移日常截图工作流。

具体客户端配置见[图床指南](/zh-cn/docs/guides/image-hosting/)。
