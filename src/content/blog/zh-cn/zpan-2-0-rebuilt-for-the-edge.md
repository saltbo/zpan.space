ZPan V1 证明了一件事：对象存储完全可以成为私人网盘的底层。应用负责目录、用户和分享，上传下载通过预签名 URL 直达 OSS、COS、S3 或 MinIO，就不必用一台昂贵服务器中转所有流量。

但“可以搭起来”和“愿意长期运行”不是一回事。传统部署仍然要求维护 VPS、反向代理、证书、数据库和进程。平时看不到这些成本，出问题时却必须立刻进入运维状态。

ZPan 2.0 因此没有只做界面升级，而是重写了运行架构。目标很明确：让控制平面可以稳定运行在 Cloudflare Workers，同时保留 Docker 这条通用自托管路径。

## S3 是数据平面，ZPan 是控制平面

用户上传文件时，ZPan 创建对象草稿和上传会话，完成权限与配额检查，再签发预签名地址。浏览器直接把文件发给 S3，上传完成后回到 ZPan 确认对象。

这种两阶段流程避免了“数据库显示文件存在，但对象只上传了一半”的状态。应用服务器也不承担文件带宽，可以把精力放在身份、目录、分享、团队和任务上。

下载在适合的场景同样使用预签名地址。存储提供商负责它擅长的传输与分发，ZPan 负责决定谁在什么时候可以拿到地址。

## 同一个应用适配 Workers 与 Node

V2 使用 Hono 构建 Web 标准的 HTTP 应用，React 提供前端，Drizzle 管理 D1 与 SQLite 家族数据库。Cloudflare Workers 通过 D1 和平台 Binding 运行；Docker 则通过 Node.js 与 SQLite 运行。

平台差异被限制在入口和适配层，文件、权限、分享等业务逻辑不会因为部署方式复制两份。这也是后续扩展到 AWS、Vercel、Netlify、Azure 和 Cloud Run 的基础。

## 文件管理重新处理了“不显眼的问题”

2.0 重做了上传、文件夹、重命名、移动、复制、搜索、回收站、预览、批量操作、进度与重名冲突。它们很难成为发布页上最吸引眼球的功能，却决定一个文件工具能否每天使用。

删除先进入回收站，上传拥有草稿与确认状态，批量操作能够反馈部分失败，替换与保留两者有明确选择。这些状态让数据库与对象存储之间的关系更可控。

## 选择 Workers 还是 Docker

如果希望尽量少维护基础设施，Workers + D1 是默认选择。它不需要常驻 VPS，适合个人与小团队。已经有服务器、需要内网运行或希望使用本地 SQLite，则选择 Docker。

两种部署都不会替你决定文件存储。站长仍需在后台添加 S3 兼容 Bucket，并配置让浏览器直传所需的 CORS。部署成功不等于文件链路已经完成，第一次上线必须实际上传并下载一个测试文件。

## V2 的起点

2.0 提供的是可以继续生长的底座。后续身份、团队、分享、图床、运营与远程下载都建立在同一个空间模型、上传会话和平台边界上。

要开始部署，请先阅读[部署方式选择](/zh-cn/docs/getting-started/choose-deployment/)，再按照 Cloudflare Workers 或 Docker 文档完成第一次端到端验证。
