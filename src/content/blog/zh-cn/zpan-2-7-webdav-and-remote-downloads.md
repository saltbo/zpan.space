文件管理产品很容易停留在“打开网页、选择文件、点击上传”这条路径上。它可以完成基本工作，但一旦接入真实使用场景，就会遇到更多入口：桌面文件管理器希望挂载目录，播放器希望直接读取媒体，用户希望把远程 URL 保存到云盘，大文件压缩又不适合挤在一次普通 HTTP 请求里。

ZPan 2.7 解决的就是这些问题。这个版本没有改变 S3 原生的基本架构，而是在控制平面之外补上了协议入口和后台执行节点，让文件可以从更多地方进入 ZPan，也可以被更多工具读取。

## 通过 WebDAV 接入已有工具

WebDAV 让 ZPan 不再局限于自己的 Web 界面。用户可以创建独立的应用密码，把个人空间或团队空间连接到支持 WebDAV 的文件管理器、同步工具与媒体播放器。

这里没有把 S3 密钥直接交给客户端。请求仍然先经过 ZPan，使用 ZPan 的账号、工作空间和权限模型，再映射到对象存储。设备丢失时，只需撤销对应应用密码，不必轮换 Bucket 凭证。

所有实例都可以使用 `/dav/` 地址。站长还可以为正式站点配置固定的 `dav.` 子域名。Cloudflare Workers 工作流能够自动协调 Custom Domain 与 Transform Rule，Docker 等部署则可以通过现有反向代理完成相同的路径改写。

## 远程下载不应该拖住主站

让 Serverless 应用直接承担长时间下载并不现实。执行时间、临时磁盘和目标站点网络条件都不稳定。因此 ZPan 2.7 把下载器设计成独立节点：主站负责任务、权限和状态，下载节点负责真正获取数据，并在完成后上传到配置的对象存储。

节点通过设备授权注册，不需要复制管理员 Cookie 或长期主密钥。它持续上报心跳、能力、速度和任务进度，站长可以从管理后台看到节点是否在线、当前负载如何，以及任务失败在什么阶段。

这种拆分也带来了更灵活的部署方式。ZPan 可以继续运行在 Cloudflare Workers，下载器则放在有临时磁盘、网络条件更合适的 VPS、NAS 或家庭服务器上。两者不需要共享文件系统。

## 从下载到播放的完整路径

WebDAV 与远程下载单独看都是功能，组合起来才形成完整工作流：

1. 用户在 ZPan 中创建远程下载任务。
2. 下载器获取文件并把结果导入 S3。
3. ZPan 更新目录与任务状态。
4. 播放器通过 WebDAV 读取同一个文件。

文件只在一个对象存储中保存，任务入口和消费入口则可以来自不同客户端。对于媒体资料、公开数据集、构建产物和远程备份，这比先下载到电脑再重新上传更直接。

## 后台任务有了明确状态

2.7 同时完善了文件夹上传、服务端压缩与解压等后台任务。耗时操作不再表现为一个长期旋转的按钮，而是拥有排队、执行、完成和失败状态。失败时可以看到原因，刷新页面也不会丢失任务记录。

Cloudflare 部署使用 Queue 执行归档任务；Node 与 Docker 使用自己的后台执行环境。无论运行在哪里，用户看到的是同一套任务模型。

## 面向运营者的补充能力

2.7 的后续补丁继续完善了 Business 授权、权益同步、Cloud Credits 与运行遥测。这些能力不会改变 Community 用户的文件路径，但能让对外运营的站点把下载、存储与流量放到统一的权益体系中管理。

## 升级与开始使用

升级后建议站长先检查后台任务与定时任务是否正常，再逐项启用 WebDAV 和下载器。WebDAV 应为每台设备创建独立密码；下载器应使用持久化 Volume 保存 Token、任务状态与缓存，并监控主机剩余空间。

如果你使用 Docker，官方 Compose 已经包含下载器服务。查看下载器日志，打开其中的设备授权地址并使用管理员批准即可。详细步骤请阅读[远程下载器文档](/zh-cn/docs/guides/remote-downloader/)与 [WebDAV 配置](/zh-cn/docs/configuration/webdav/)。
