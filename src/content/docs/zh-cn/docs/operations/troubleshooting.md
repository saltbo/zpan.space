按层排查问题：浏览器、ZPan API、数据库、对象存储，最后再检查邮件和下载节点等可选服务。

## 快速定位

1. 打开 `/api/health`，并查看故障发生在同一 UTC 时间的部署日志。
2. 在浏览器无痕窗口复现，排除旧会话和扩展影响。
3. 打开 Network 面板，记录第一个失败请求、状态码和响应内容。
4. 确认部署版本，以及故障前最后一次配置变更。

| 现象 | 优先检查 | 第一项核对 |
| --- | --- | --- |
| 登录后循环跳转 | 公开地址或代理 | `BETTER_AUTH_URL`、HTTPS、转发 Header |
| 存储测试成功但上传失败 | 浏览器到 Bucket | CORS 与公网 Endpoint |
| 上传结束却没有文件 | 确认请求或数据库 | API 日志和数据库错误 |
| 下载返回 403 | 预签名或时间 | 存储策略、Region、服务器时钟 |
| OAuth 回调被拒绝 | 服务商客户端 | 完整回调地址与密钥 |
| 收不到邮件 | SMTP | 发件策略、TLS、服务商日志 |
| 远程任务不启动 | 下载节点 | 在线状态与设备授权 |

原因不明时不要反复执行破坏性操作。手工修复数据前先保存日志并备份数据库。求助时应提供最小复现、部署平台、ZPan 版本、脱敏配置、浏览器错误和相关服务端日志。

## 日志与运行状态

健康检查必须与平台日志一起使用。`GET /api/health` 只能证明应用进程能够响应，不能证明浏览器上传、邮件、定时任务或每个存储后端都正常。

- 从托管平台外部监控 `/api/health`，连续非 2xx 或延迟异常时告警。
- 按明确周期保存应用、反向代理、数据库和对象存储日志。
- 关注重启循环、数据库错误、登录失败、签名错误、定时任务失败和异常的 4xx/5xx 比例。
- 每次发布或基础设施变更后执行一次小文件上传与下载探测。

Docker 可先查看 `docker compose ps` 和 `docker compose logs --tail=200 zpan`；Cloudflare 使用 Workers 可观测性及 D1/R2 控制台；其他平台查看自己的函数或容器日志。

调查时记录 UTC 时间、请求路径、用户或空间、部署版本和请求标识。不要公开 Cookie、Authorization Header、OAuth Secret、存储凭据或预签名 URL。
