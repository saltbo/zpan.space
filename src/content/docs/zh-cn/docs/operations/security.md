title: "安全加固"
description: 使用稳定密钥、HTTPS、最小存储权限和受控注册加固公网 ZPan 实例。
---

应把 ZPan 当作能够访问用户文件的公网应用来防护。基础原则是最小权限、稳定密钥、全站 HTTPS 和可执行的升级流程。

## 安全基线

- 为 `BETTER_AUTH_SECRET` 生成足够长的随机值，保持稳定且不要提交到仓库。
- 公开地址必须使用 HTTPS，并让反向代理正确传递原始 Host 和 Scheme。
- 条件允许时，以非 root 用户运行 ZPan 和数据库。
- 数据库、存储控制台和下载节点管理端口只允许私网访问。
- 为 ZPan 创建只限目标 Bucket 和所需对象操作的独立密钥。
- 生产环境 CORS 只允许实际的 ZPan Origin，不要使用 `*`。
- 不需要公开注册时将其关闭；公开注册时启用验证码。
- 只保留可信且实际使用的邮件、OAuth 或 OIDC 登录方式。

定期检查管理员、团队 Owner、API Key、OAuth 客户端、人工配额和在线下载节点。审计日志有助于追踪事件，但不能代替告警和备份。

密钥泄露后，应先在签发方轮换、更新 ZPan 并确认新密钥可用，再撤销旧值。轮换 `BETTER_AUTH_SECRET` 可能使会话和派生凭据失效，需要安排维护窗口。存储密钥泄露时，还要检查服务商访问日志和 Bucket 变更。
